# V3.md — Milestone 3 "Practice": journal, SRS, reverse quiz, weekly review

Derived from .planning/VISION-1M.md §4/§9 (v3 = the retention machine). Raw human go: "run v3" (2026-07-10).

## Why this milestone

A reference library is visited twice; a practice with accruing personal data is visited daily. v3 turns the app from library into practice: your decisions (journal), your mastery (SRS), your recall (quiz), your follow-through (weekly review). Everything local, everything offline, no account — soul constraints from VISION-1M §0 are BLOCKING contract terms.

## Part A — Infrastructure first (prerequisites, logged since v2 reviews)

### A1. PDB_STORE — one shared, versioned storage module (js/store.js)

- Single hardened localStorage gateway all features use: `PDB_STORE.get(key)`, `set(key, value)`, with (a) try/catch degradation to in-memory (never throws; private-mode safe), (b) fresh read-through per call (the favorites.js multi-tab lesson), (c) a stored `pdb.schemaVersion` and a migration hook (`migrate(fromVersion)`) so future shape changes never silently drop user data (the v1 researcher finding).
- Migrate EXISTING keys behind it without data loss: `pdb.favorites`, `pdb.applied`, `pdb.theme`, `pdb.testDate`. Existing users' favorites/streak/theme MUST survive the upgrade byte-for-byte. New v3 keys (journal, srs, quiz) go through it from day one.
- favorites.js/daily.js/app.js refactored to call PDB_STORE; their public APIs (PDB_FAV.isFavorite/toggle/all, PDB_DAILY.*) stay byte-compatible so all existing suites keep passing with at most storage-plumbing test updates.

### A2. Persona-tabs extraction (js/persona-tabs.js)

- Extract the ARIA tab widget out of card.js into `PDB_PERSONA_TABS.render(examples, featured, scope, onSelect)`; card.js becomes the pure card orchestrator calling it. All existing behavior preserved (roving tabindex, arrows/Home/End, panel tabindex=0, click focus, per-instance ids).
- The `onSelect(personaId, index)` hook is the point: v3+ features (quiz, SRS, future analytics) can observe persona engagement. In v3 the hook records "personas explored" locally (used by weekly review copy); no other consumer yet.

## Part B — Decision journal

The Farnam Street decision-journal template, verified in RESEARCH.md, as a first-class feature.

- **Entry fields** (all optional except situation + expected outcome): situation/context; the decision frame ("what am I actually deciding?"); options seriously considered incl. rejected ones + why; expected outcome + confidence (0–100% slider or 5-step); how I feel/state (energy, mood — one tap each); linked framework(s) (picker from the 74, optional); review date (default +90 days, choosable +30/+90/+180).
- **Views**: journal list (newest first, status chips: open / review-due / closed), entry detail, new-entry form. Entry point: a "Practice" bottom tab (see Part F) AND a "journal this decision" affordance on every framework card.
- **Close the loop**: when the review date arrives, the entry surfaces (weekly review, Part E) asking: what actually happened? outcome vs expectation, and "what did the gap teach you?" — the anti-hindsight-bias mechanic that motivated the feature.
- **Export**: one tap → plain-text/markdown export of any entry or all entries (share sheet / clipboard) — user's data is portable, reinforcing the on-device trust story.
- Storage: `pdb.journal` via PDB_STORE. No caps (no Pro tier in v3). Concrete-stakes placeholder copy in the form teaches good entries (numbers, deadlines, named tensions — same bar as the examples).

## Part C — SRS engine (framework mastery)

- Per-framework SRS state keyed by framework id via PDB_STORE (`pdb.srs`): `{lastReviewedAt, intervalDays, due, easeStep}`. Simple graded intervals (Leitner-style ladder, e.g. 1 → 4 → 12 → 30 → 90 days; "again" resets to 1, "got it" advances) — NOT full SM-2; deterministic and explainable. Injectable date source reused from PDB_DAILY.today() (single clock).
- **Grading moments**: after reading the daily card ("Did you know this one? again / got it"), and after each quiz answer (Part D auto-grades: wrong = again, right = advance).
- **Today screen integration**: Today becomes two-deck — the daily card (rotation, unchanged, streak preserved) PLUS a "due for review" shelf (0–3 due frameworks max per day, oldest-due first) with one-tap grade. Rotation/streak logic (B16–B18) must NOT regress; SRS is additive.
- All 74 frameworks start unscheduled; a framework enters SRS the first time it's graded (daily card or quiz). Empty states designed ("nothing due — come back tomorrow").

## Part D — Reverse quiz ("Which framework fits?")

- Quiz round = 5 questions. Each question: one scenario drawn from the 370 (scenario text WITHOUT naming its framework) + 4 answer options (the correct framework + 3 plausible distractors from the same/nearby category), tap to answer, immediate feedback with the framework's essence + a link to the card.
- Question selection prioritizes: SRS-due frameworks first, then unseen-by-quiz, then random. Scenario chosen randomly among the framework's 5 personas — seeded deterministically per (framework, date) so tests can pin it (no Math.random in product code paths that tests assert; reuse the injectable clock/seed pattern).
- Feeds SRS (Part C). Tracks per-round score locally (`pdb.quiz`): rounds played, accuracy — shown on the Practice tab. No leaderboards, no global anything.
- Guard: never show a scenario as a question when its framework card was open in the same session's last N views if avoidable (cheap recency check) — keeps it a recall test, not a memory-of-30-seconds-ago test.

## Part E — Weekly review ritual

- A "Weekly review" card on the Practice tab that activates when there's material: journal entries whose review date arrived, plus a 60-second recap (this week: decisions journaled, frameworks practiced, quiz accuracy, personas explored via A2 hook).
- Flow: for each due journal entry → record actual outcome vs expected + confidence calibration note → entry closes. End screen: one gentle prompt ("One decision you're avoiding this week?") linking to new-entry form.
- No pressure mechanics: it's available, never nagging (notifications are v5; nothing in v3 pings).

## Part F — Navigation & IA change

- Bottom tabs become: **Situations · Browse · Today · Practice** (Favorites moves INTO Practice as a section, or into Browse — planner/contract decides with reachability preserved: every v1/v2 reachability guarantee (≤3 taps, B10) must still hold, and favorites data/UX must not regress).
- Practice tab = journal list + quiz entry + weekly review + favorites (if placed here) + a small stats strip (streak, rounds, accuracy, entries).
